# Content Intelligence — пилот «шины и ремонт»

Разведка контента: что залетает на RU/US-каналах × чего нет в казнете = брифы.

**Деливерабл пилота:** `output/gap_analysis.png` (график для слайда) +
`output/briefs/brief_01.md … brief_10.md`. Папка с транскриптами — не деливерабл.

## Архитектура

Пайплайн из 7 независимых этапов. Каждый этап — самостоятельный скрипт:
читает вход с диска, пишет выход на диск, кэширует всё дорогое. Упал —
перезапускаешь только его, сделанное не переделывается.

```
config/channels.yaml (каналы-доноры, вручную)
        │
        ▼
[01 harvest]    yt-dlp: метаданные видео за N месяцев ──▶ data/raw/videos.csv
        ▼
[02 score]      pandas: vpd, outlier_score, is_hit ─────▶ data/scored/videos.csv
        ▼
[03 subtitles]  yt-dlp: авто-субтитры для всех ─────────▶ data/subs/*.txt
        ▼
[04 transcribe] Whisper large-v3 ТОЛЬКО по хитам ───────▶ data/transcripts/*.txt
        ▼
[05 topics]     LLM: тема каждого видео + кластеризация ▶ data/clusters.json, topics.csv
        ▼
[06 gap]        LLM + поиск YouTube: спрос RU vs KZ ────▶ output/gap_table.csv, gap_analysis.png
        ▼
[07 briefs]     LLM: контент-брифы по топ-темам ────────▶ output/briefs/brief_01..10.md
```

### Сигнал: как ищутся «залетающие» темы

Абсолютные просмотры сравнивать нельзя — у каналов разный размер. Поэтому:

- `vpd = view_count / days_live` — просмотры в день жизни ролика
  (days_live обрезается снизу единицей, чтобы свежие видео не взрывали метрику);
- `outlier_score = vpd / медиана vpd ЭТОГО ЖЕ канала` — во сколько раз ролик
  обгоняет собственную норму канала. Это классический outlier detection
  относительно baseline канала: скор 4.0 у канала-миллионника и у канала на
  10к подписчиков означает одно и то же — «тема выстрелила»;
- `outlier_score ≥ 2.0` → хит. Только хиты идут в дорогую транскрипцию.

### Дыры в рынке (gap-анализ)

Для каждого кластера тем LLM генерирует поисковые запросы (RU + KZ/казахские),
`yt-dlp ytsearch` собирает выдачу, каждое найденное видео классифицируется
«казахстанское / нет» (эвристика по маркерам + LLM-классификатор). Дальше:

```
gap_score = ru_views / (kz_views + 1000)
```

Просмотры темы в RU против просмотров той же темы в казнете; +1000 в
знаменателе — сглаживание, чтобы пустой казнет не давал деление на ноль.
Большой gap_score = спрос доказан, а локального предложения нет.

### ML-компоненты

**ASR — faster-whisper large-v3.** Батчевый инференс (3–4× быстрее), на 4 ГБ
VRAM работает в `int8_float16` (~3 ГБ). Язык определяется автоматически на
каждом видео (среди доноров есть англоязычные каналы); при форсированном `ru`
дополнительно подключается доменный глоссарий через `initial_prompt` —
заметно поднимает точность на профлексике («развал-схождение», «кварцвинил»).
Транскрибируются только хиты: ASR — самый дорогой этап, скоринг на метаданных
режет объём работы на порядок.

**LLM — два уровня через Groq API.** Дневные лимиты бесплатного Groq считаются
по каждой модели отдельно, поэтому работа разделена:

- `openai/gpt-oss-120b` — «умственное»: кластеризация тем, генерация поисковых
  запросов, брифы. Это reasoning-модель, поэтому в вызовах ограничен
  `reasoning_effort: low` — иначе на больших задачах она тратит весь бюджет
  токенов на внутренние рассуждения и возвращает пустой ответ;
- `llama-3.1-8b-instant` — «массовое»: тема каждого из ~300 видео,
  KZ-классификация сотен результатов поиска.

**Инженерия под жёсткие лимиты бесплатного тарифа** (у gpt-oss-120b —
8000 токенов в минуту на ВЕСЬ запрос, вход+выход):

- каждый LLM-вызов кэшируется на диск по sha256(модель+промпт+параметры) —
  перезапуск этапа не тратит лимиты и делает результаты воспроизводимыми;
- кластеризация оперирует порядковыми номерами видео вместо 11-символьных
  video_id — это единственный способ уместить ~140 тем и ответ модели
  в минутный лимит;
- битый JSON от модели ретраится с удвоенным `max_tokens` (частая причина —
  обрезка по лимиту), обрезанные ответы не кэшируются;
- расход токенов пишется в `logs/llm_usage.jsonl` — по нему экстраполируется
  стоимость масштабирования на платном тарифе.

**Почему LLM, а не классика (эмбеддинги + HDBSCAN, LDA и т.п.)?** На объёме
пилота (~300 видео, 15–25 кластеров) LLM-кластеризация дешевле в разработке,
даёт сразу человекочитаемые названия кластеров и «понимает» доменную близость
тем («шипы против липучки» ≈ «выбор зимней резины») без подбора гиперпараметров.
На «точке ноль» (500–1000+ видео) кластеризацию имеет смысл перевести на
эмбеддинги, LLM оставить только для нейминга кластеров.

### Тесты

`tests/` — 73 pytest-теста без сети и без LLM (всё через моки и tmp_path):
формулы скоринга и gap, JSON-ремонт ответов модели, дисковый кэш, ретраи,
оркестратор. Отдельный тест закрепляет правовую рамку контрактом: в промпты
брифов не могут попасть video_id, URL и названия чужих видео.

```bash
python -m pytest tests -q
```

## ⚠️ Правовая рамка (обязательна)

Транскрипт чужого видео — **сырьё для ресёрча, а не черновик статьи**.

- МОЖНО: извлекать факты, темы, вопросы аудитории, структуру спроса, терминологию.
- НЕЛЬЗЯ: публиковать расшифровку, рерайтить её «своими словами», гонять через
  ИИ-синонимайзер и выкладывать.

Папка `data/transcripts/` (и вся `data/`) **не является контентом и не уходит
в паблик** — она в `.gitignore` и не должна попадать в репозиторий, облако,
презентации. Брифы генерируются с явным запретом на пересказ и цитаты.

## Требования

- Python 3.11+
- **ffmpeg в PATH** — нужен yt-dlp для конвертации субтитров и извлечения аудио
  (Windows: `winget install ffmpeg` или скачать с ffmpeg.org)
- Бесплатный ключ Groq API — https://console.groq.com/keys
- Для этапа 04: GPU желателен. Без GPU поставь в `config/settings.yaml`
  `transcribe.model: medium` (large-v3 на CPU — примерно реалтайм и медленнее).

## Установка

```bash
cd content-intel
python -m venv .venv
.venv\Scripts\activate          # Windows
pip install -U -r requirements.txt
pip install -U yt-dlp           # обновлять ПЕРЕД КАЖДЫМ запуском — YouTube регулярно ломает парсеры

copy .env.example .env          # и вписать GROQ_API_KEY (бесплатный, console.groq.com/keys)
```

## Перед запуском

1. Заполни `config/channels.yaml` — 10–15 каналов-доноров (RU/BY/US, шины и
   ремонт). Это ручная работа (~30 минут) и она критична для качества.
2. Проверь пороги в `config/settings.yaml` (лимиты, порог хита, модель Whisper).

## Запуск одним кликом

- **`run_pipeline.bat`** (двойной клик) — локальный запуск: сам создаёт venv,
  ставит зависимости, обновляет yt-dlp и прогоняет все 7 этапов. Рекомендуемый
  способ — Whisper видит GPU, если он есть.
- **`run_docker.bat`** — то же в Docker (образ собирается сам; ffmpeg внутри).
  Минус: Whisper в контейнере работает на CPU. Конфиги/данные/результаты
  монтируются с хоста, правка channels.yaml пересборки не требует.

Оба принимают аргументы оркестратора: `run_pipeline.bat --to 4` (без LLM-ключа,
только сбор), `run_pipeline.bat --from 5` (продолжить после падения/лимита),
`--only 6` (один этап). Перед стартом идёт быстрая проверка: каналы заполнены,
ключ Groq на месте (для этапов 5-7).

## Запуск по этапам (вручную)

Каждый скрипт самостоятельный: читает результат предыдущего с диска.
Если этап упал — чинишь и перезапускаешь только его, всё закэшировано
(метаданные, субтитры, транскрипты, вызовы LLM, поисковая выдача).

```bash
python src/01_harvest.py      # метаданные каналов -> data/raw/videos.csv
python src/02_score.py        # vpd, outlier_score, is_hit -> data/scored/videos.csv
python src/03_subtitles.py    # авто-субтитры для всех -> data/subs/*.txt
python src/04_transcribe.py   # Whisper только по хитам -> data/transcripts/*
python src/05_topics.py       # темы + кластеризация -> data/topics.csv
python src/06_gap.py          # ⭐ gap-анализ -> output/gap_table.csv + gap_analysis.png
python src/07_briefs.py       # 10 брифов -> output/briefs/
```

## Проверка казахского языка (обязательный пункт ТЗ)

Whisper на казахском заметно слабее, чем на русском. Если в донорах есть
казахоязычные ролики — прогони 2–3 и оцени глазами:

```bash
python src/04_transcribe.py --language kk --video <video_id>
```

Результат — в `data/transcripts/<video_id>.txt`. Если качество не годится,
зафиксируй это в отчёте как ограничение; альтернатива — модели ISSAI (Soyle).
Выяснить сейчас, а не через месяц.

## LLM: бесплатный Groq API

- Модель задаётся в `settings.yaml` (`models.llm`), актуальный список —
  https://console.groq.com/docs/models
- **Бесплатный тариф ограничен по запросам в минуту И по токенам в день.**
  Пайплайн это переживает: все вызовы кэшируются на диске, поэтому если этап
  упал на дневном лимите — просто перезапусти его на следующий день, он
  продолжит с места остановки, не тратя лимиты на уже сделанное.
- SDK сам ретраит 429; пауза между вызовами — `models.sleep_between_calls`.

## Стоимость и время (для экстраполяции на «точку ноль»)

- Groq бесплатный, но расход токенов всё равно пишется в
  `logs/llm_usage.jsonl` — по нему считается стоимость масштабирования
  на платном тарифе / другом провайдере. Суммарный расход:

  ```bash
  python -c "import json;rs=[json.loads(l) for l in open('logs/llm_usage.jsonl',encoding='utf-8')];print(f\"in={sum(r['input_tokens'] for r in rs):,} out={sum(r['output_tokens'] for r in rs):,}\")"
  ```

- Время каждого этапа — в конце соответствующего `logs/0N_*.log`.
- Экстраполяция: цифры пилота × объём (~500–1000 видео «точка ноль» + ежедневный
  обход). Для «точки ноль» дневных лимитов бесплатного Groq может не хватить —
  заложить в оценку платный тариф Groq или другого провайдера.

## Если YouTube банит / bot-detection

- `--sleep-requests 1` уже включён (settings.yaml). При бане — подожди и увеличь до 2–3.
- Обнови yt-dlp: `pip install -U yt-dlp`.
- Крайняя мера — куки браузера: yt-dlp `--cookies-from-browser chrome`
  (в скриптах не включено намеренно; добавляй руками при необходимости).
